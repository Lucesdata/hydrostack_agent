# Fase 0.5 — Ingesta incremental

> Sub-fase de implementación (CÓDIGO permitido). Depende de [0.2-spec-ingesta.md](0.2-spec-ingesta.md) (diseño) y [0.4-capa-normalizada.md](0.4-capa-normalizada.md) (capa canónica). Decisiones D14, D21 cerradas en 0.2; D21a, D28, D29 cerradas en esta fase.

## 0. Resumen

Cierra el ciclo ELT contra la API real: keyset incremental sobre SECOP II + sweep periódico de registros sin watermark + transformación post-ingesta. Todo orquestado por un CLI único.

Esta sub-fase **no introduce esquema nuevo**: el control (`sync_log`) ya estaba en 0.3. Lo que entrega es el cableado E2E + la pasada 2 del incremental (D21a) que faltaba.

## 1. Arquitectura

```
┌──────────────────────────────────────────────────────────────────────┐
│                       scripts/run-ingest.ts                          │
│                                                                      │
│  por fuente {procesos, contratos}:                                   │
│                                                                      │
│    PASADA 1 (keyset, D14) ─→ ingestSource()                          │
│      lee sync_log.watermark_to anterior                              │
│      ventana: [watermark - 1d, ∞), keyset (watermark, id)            │
│      sink dedup por payload_hash                                     │
│      escribe sync_log (status, watermark_to)                         │
│                                                                      │
│    PASADA 2 (sweep, D21a) ─→ sweepWithoutWatermark()  [solo contratos]│
│      WHERE ultima_actualizacion IS NULL ORDER BY id_contrato         │
│      sink dedup por payload_hash                                     │
│      NO mueve watermark (estos registros no tienen timestamp)        │
│                                                                      │
│  TRANSFORM ─→ runTransform()                                         │
│    reconstruye TODA la canónica desde raw_record                     │
│    idempotente; upsert por clave natural                             │
└──────────────────────────────────────────────────────────────────────┘
```

**Por qué dos pasadas:** la fuente de contratos trae `ultima_actualizacion` solo en ≈57% de los registros (medido en 0.2.2). El keyset puro nunca ve el otro 43%. El sweep los recoge por id nativo; el hash del payload evita reprocesar lo que no cambió.

**Por qué el transform reconstruye todo:** simplicidad + consistencia. La foto canónica es siempre correcta; nunca queda en un estado intermedio "el batch nuevo se aplicó pero los upserts cruzados quedaron pendientes". A volumen actual el costo es despreciable; en fase 1 se optimiza con filtro por `batch_id`.

## 2. CLI: `npm run db:ingest`

```bash
npm run db:ingest                          # keyset + sweep + transform (ambas fuentes)
npm run db:ingest -- --skip-transform      # solo aterrizar a raw_record
npm run db:ingest -- --source procesos     # una sola fuente
npm run db:ingest -- --source contratos
npm run db:ingest -- --sweep-only          # solo D21a (sin keyset)
npm run db:ingest -- --json                # output JSON estructurado
npm run db:ingest -- --page-size 500 --margin-days 1 --max-pages 50
```

Flags:

| Flag | Default | Significado |
|------|---------|-------------|
| `--source` | `both` | `procesos` \| `contratos` \| `both` |
| `--skip-transform` | off | no llamar `runTransform()` después |
| `--sweep-only` | off | omitir pasada 1 (debug) |
| `--json` | off | output estructurado (`stdout`) |
| `--page-size` | 1000 | filas por página SODA |
| `--margin-days` | 1 | solape D14 sobre el watermark |
| `--max-pages` | 10000 | tope de páginas por pasada |

Salida humana:

```
contratos (secop_ii_contratos):

  sweep:   0 ingested · 100 scanned · 2 págs [TRUNCADO]

```

## 4. Decisiones cerradas (0.5)

### D21a. Sweep periódico para registros sin watermark — **confirmada (a)**
**Decisión:** sweep full por id nativo, sin estado incremental del barrido. El `payload_hash` evita reprocesar lo que no cambió, así que un sweep redundante es barato. Para fase 0 es suficiente.

**Justificación:** la muestra trae ≈215 contratos sin `ultima_actualizacion` (43%). Cada sweep los recorre por id ascendente; el hash en el sink descarta los inmutables. Cuando el volumen lo justifique (fase 1+) se migra a (b) con cursor por id persistido en `sync_log`.

**No actualiza `watermark_to`:** los registros sin `ultima_actualizacion` no son "nuevos" desde la perspectiva temporal; son "nunca-vistos-por-watermark". Hacerlos avanzar el watermark contaminaría la cota inferior del keyset.

### D28. Pipeline ingesta → transform — **confirmada (secuencial, sin filtro de batch)**
**Decisión:** un solo CLI corre ingesta + transform en serie. El transform NO filtra por `batch_id`: reconstruye toda la canónica desde `raw_record`. Idempotente por diseño (0.4).

**Justificación:** simplicidad + garantía. La foto canónica es siempre consistente, incluso si la ingesta corre parcial. El costo de reconstruir 698+699 registros es ~3 s; se optimiza cuando duela (fase 1+).

### D29. Backfill vs incremental — **confirmada (misma ruta, mismo CLI)**
**Decisión:** no hay flag `--mode backfill` separado. El backfill es simplemente la 1ª corrida (sin watermark previo) con `--max-pages` grande. El incremental son las siguientes (con watermark heredado).

**Justificación:** si el backfill se desea en ventanas trimestrales (D9), eso se logra controlando `--max-pages` y dejando que el watermark avance. No hace falta lógica nueva.

## 6. Detalles de diseño que terminaron importando

### Pureza del bucle, IO inyectado

`runIngest` y `runSweep` viven en `runIngest.ts` con `SodaFetcher` y `RawSink` como parámetros. Eso permite los 5 tests del sweep + 6 del keyset (puros, sin red ni DB).

`dbIngest.ts` solo cabléa los adaptadores reales (`sodaFetchPage` + `makeDbSink`). No tiene lógica de control de flujo.

### Estado `'partial'` en `sync_log`

Cuando `reachedMaxPages = true`, el watermark sí se persiste hasta donde llegó, pero el `status` se marca `'partial'`. La siguiente corrida sigue tomando ese watermark (porque `readLastWatermark()` filtra `IN ('ok', 'partial')`), así que el flujo es: corrida grande → varias parciales → eventualmente `'ok'` cuando la fuente se queda sin filas nuevas.

### Pool cleanup

El CLI llama `pool.end()` en `finally` del catch global, no solo en el `main()` exitoso. Sin esto, errores en producción dejarían conexiones zombi.

### `pool.end()` después de error

```ts
main().catch((e) => {
  process.stderr.write(`✖ ${...}\n`);
  pool.end().catch(() => {}).finally(() => process.exit(1));
});
```

El `.catch(()=>{})` previene un unhandled rejection si el pool ya estaba cerrado. El `.finally(process.exit)` garantiza que el proceso termina aunque `pool.end()` cuelgue.
