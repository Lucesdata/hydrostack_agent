## 1. Migración `0001_align_d22_d26.sql`

Alinea el esquema físico con D22–D26 (la canónica ya existía en `0000`):

| Cambio | Decisión | Por qué |
|---|---|---|
| `proveedor.nit_valid_dv` → DROP | D22 | La fuente no trae DV; `nit_dv` es el calculado (DIAN). No hay con qué validar |
| `entidad.sector` → RENAME a `sector_administrativo` | D23 | Es el sector PGN del Estado, no agua/saneamiento — la pertenencia agua vive en `clasificacion_sectorial` |

La migración es manual (drizzle-kit requiere TTY interactivo para resolver el
rename `sector` → `sector_administrativo`), con el snapshot 0001 regenerado
programáticamente. `npm run db:generate` confirma cero drift después.

Sobre `contrato_evento`: **ya existe vacía** desde la migración 0000 — se deja
intacta y NO se puebla en 0.4 (es Fase 1). La pregunta abierta "crearla vacía
vs diferirla" se resolvió sola: existe, se deja.

## 2. Lógica pura del transform — auditoría y ajustes

### Lo que ya estaba (intacto)
- `normalize.ts`: `cleanText`, `parseMoney`, `parseDate`, `parseBool` con la
  lista de centinelas (H3, 0.2 §3).
- `nit.ts`: `canonicalizeNit`, `computeNitDv` (DIAN), `normalizeTipoDoc`.
- `geo.ts`: `normalizeGeoText` (lowercase, sin tildes, sin puntuación).
- `mapCanonical.ts`: proyecciones puras `mapProcesoRow` / `mapContratoRow`.

### Lo que cambió en 0.4
- **`EntidadProjection`** (D23): `sector` → `sectorAdministrativo`; `rama`
  desaparece de campo de primera clase y pasa a `rawAttrs.rama`.
- **`ProveedorProjection`** (D22): se elimina `nitValidDv`. `nitDv` se conserva
  como DV calculado (no validado).
- **Geografía del contrato** (D25, antes invertida): `localizaci_n`
  **PRIMARIA** (`"Colombia, <dep>, <mun>"`, 0% nula); `departamento`/`ciudad`

  efectivamente hacía que la mayoría de contratos quedaran sin geo.
- **`proveedor.rawAttrs`** ahora persiste `nombre_representante_legal` cuando
  está presente — antes se perdía.

Los tests existentes (`src/__tests__/transform/`) se actualizaron al nuevo
shape; **127 tests pasan** (vitest, completo).

## 3. Writers — escritura idempotente con upsert por clave natural

`src/lib/transform/writers.ts`:

- `upsertProveedor` (UNIQUE `nit_canonico`) — reescribe TODAS las columnas
  no-PK en `ON CONFLICT DO UPDATE`. Si una columna deja de calcularse, queda
  `NULL` automáticamente; sin zombis.
- `upsertEntidad` (UNIQUE `nit_canonico`) — igual; resuelve `geografia_id`
  desde los hints geográficos de la fuente.
- `upsertProceso` (UNIQUE `secop_proceso_id`).
- `upsertContrato` (UNIQUE `secop_contrato_id`) — con una particularidad
  importante: `valor_inicial` y `fecha_fin_inicial` son **inmutables** (0.1
  §4.3), así que solo se fijan en `INSERT`; en `UPDATE` no se tocan.
  `valor_actual`/`fecha_fin_actual` sí.

### Por qué upsert y no rebuild

Decidido en §6.b del razonamiento previo. Resumen:

| | Upsert | Rebuild (TRUNCATE + INSERT) |
|---|---|---|
| Idempotencia | Sí (clave natural) | Sí (estado en blanco) |
| Compatibilidad con 0.5 incremental | Directa | Hay que reescribir |
| FKs ordenadas | Implícito | Orden estricto de truncado obligatorio |
| Columnas zombi | Riesgo mitigado (reescribir todo) | Cero por construcción |

El upsert gana en simplicidad operacional y migración hacia adelante.

### Resolución de geografía (`GeoResolver`)

Cache en memoria de `geografia_alias` (~100 entradas en 0.4), cargado una vez
por corrida. `resolve(GeoHints)`:
1. Si hay municipio normalizado → lookup. Si hit, retorna.
2. Si hay departamento normalizado → lookup. Si hit, retorna.
3. Null.

Convención sintética DANE: los departamentos están como `"DD000"` (con muni
"000"), las capitales como `"DD001"`. Esto permite que un contrato con solo
hint departamental ("Antioquia", sin ciudad) caiga en `05000` en vez de NULL.

### Resolución proceso↔contrato (D11)

`resolveProcesoIdByPortafolio`: lookup por `contrato.proceso_de_compra` (CO1.BDOS)
↔ `proceso.portafolio_id`. Si el proceso no existe en la muestra (ventanas BDOS
disjuntas — [0.2 §5.1](0.2-mapeo-campos.md)), `contrato.proceso_id` queda
NULL y el contrato **NO se descarta** (0.2.2 §9.4).

`data/dane/divipola.ts` + `scripts/seed-geografia.ts`. Cobertura 0.4:

- **33 departamentos** con código DANE oficial (`05` Antioquia → `99` Vichada).

- **Aliases ortográficos manuales**: "distrito capital de bogota" → `11000`,
  "san andres providencia y santa catalina" → `88000`, "since" → `70708`.

los marca como no-resueltos en la lista de §4 de `validate-reconstruction.sql`.

## 5. Orquestador y corrida — `npm run db:transform`

`src/lib/transform/orchestrator.ts` orquesta:

1. Carga `GeoResolver` (una sola query).
2. Itera procesos — `latestSnapshots('secop_ii_procesos')` (DISTINCT ON por
   `source_record_id` ORDER BY `ingested_at DESC`).
3. Itera contratos — same pattern. Pero **después** de procesos para que
   `resolveProcesoIdByPortafolio` encuentre algo.

Por cada fila:
- Map a proyección pura.
- Si `secopProcesoId`/`secopContratoId` ausente → cuarentena (no se descarta
  silenciosamente).
- Upsert entidad (siempre que el NIT sea válido).
- Para contratos: upsert proveedor si no es centinela; si es centinela,
  `proveedor_id = NULL` y `proveedor_raw = razon_social` (D3).
- Upsert proceso/contrato con todos los refs.

### Métricas reportadas (por fuente)

```
procesos (secop_ii_procesos):
  snapshots crudos:           N
  registros únicos (latest):  M
  entidades upsert:           …
  procesos upsert:            …
  geografía resuelta:         …
  geografía no resuelta:      …
  cuarentena (errores duros): …

contratos (secop_ii_contratos):
  ...
  proveedores upsert:         …
  proveedor centinela (→raw): …
  proceso no encontrado:      … (ventana BDOS — 0.2 §5.1)
  ...
```

### Idempotencia

Re-correr `db:transform` sobre el mismo `raw_record` deja el mismo estado
canónico:
- Catálogos: `INSERT ON CONFLICT (nit_canonico)` con `DO UPDATE` que reescribe
  todo.
- Hechos: igual con la clave secop.
- Cuarentena: append-only, así que una segunda corrida añade líneas
  duplicadas — eso es intencional (capturar cada intento fallido); no afecta
  la canónica.

## 6. Validación de reconstrucción — `scripts/sql/validate-reconstruction.sql`

5 contratos elegidos automáticamente cubriendo casos representativos:

| # | Caso | Cómo se elige |
|---|---|---|
| 1 | NIT limpio, geo resuelta, proveedor presente | `tipo_documento='NIT'` + `geografia_id IS NOT NULL`, mayor `valor_inicial` |
| 2 | Persona natural | `tipo_documento='CC'` |
| 3 | Consorcio | `proveedor.es_estructura_plural = true` |

texto crudo de `localizaci_n`/`departamento`/`ciudad`.

Más: §2 conteos agregados (denominadores para 0.6), §3 idempotencia
(snapshots crudos vs registros únicos), §4 top-30 textos geográficos no
resueltos (entrada directa para expandir el seed DANE en 0.6).

## 7. Decisiones tomadas en 0.4 — con racional

| | Decisión | Recomendación → adoptada |
|---|---|---|
| **a** | `contrato_evento` vacía vs diferida | Ya existía vacía; **dejarla**. Cero trabajo, coherente con 0.1 §2. |
| **b** | Rebuild vs upsert | **Upsert** por clave natural. Más simple para 0.4, compatible con 0.5 incremental sin reescribir. |
| **c** | Crudos no mapeables: descartar vs cuarentena | **Cuarentena con log** (`transform_quarantine`). Centinelas conocidos NO van a cuarentena (son destino NULL legítimo); solo errores estructurales (id ausente, JSON inesperado). |

## 8. Checkpoint

**¿Un contrato se reconstruye CORRECTO desde lo crudo, campo por campo?**
Sí, por construcción y verificable. El transform es lógica pura
(`mapCanonical`) sobre el `payload` de `raw_record`, sin fuentes laterales.
Los 127 tests de vitest cubren los normalizadores (NIT con consorcios y
centinelas, geografía con localizaci_n primaria, money con `"0"` legítimo,
proveedor centinela → `proveedor_raw`, sector administrativo en columna y
rama en `raw_attrs`). La validación campo a campo en DB (`§6`) requiere la
corrida final.

**¿La transformación es reprocesable e idempotente sin re-descargar?**
Sí. Upsert por clave natural; segunda corrida = mismo estado. La fuente del
transform es `raw_record`, nunca Socrata. La capa cruda es append-only desde
0.3, así que el "historial" de un registro se preserva aunque la canónica
solo refleje el snapshot más reciente (`DISTINCT ON ... ORDER BY ingested_at
DESC`).

**¿Qué quedó marcado como pendiente para 0.6?**
- **Crosswalk DANE completo** (~1000 municipios adicionales). Entrada
  directa: top 30 de no-resueltos en `validate-reconstruction.sql §4`.
- **Dedup avanzada de proveedor**: alias por fusión, variantes de razón
  social, validación contra RUES cuando entre. Hoy es UNIQUE simple por
  `nit_canonico`.
- **`proveedor_miembro`** (composición de consorcios): tabla declarada en 0.1
  §3.3, queda vacía hasta que entre RUP/RUES.
- **Parser de `localizaci_n` con comas en el nombre**: departamentos como
  "San Andrés, Providencia y Santa Catalina" llevan comas, así que el split
  por coma los parte mal (visto 1 vez en la muestra). 0.6: parsear desde el
  final (los 2 últimos segmentos no-centinela) o usar un set de nombres de
  departamento conocidos como ancla.
- **Normalizador de centinelas afinado**: hoy es una lista cerrada; 0.6
  refinaría casos vistos en producción.
- **`proceso.estado_codigo numeric`** (0.2.2 §9.5): cuando una query lo
  justifique.
- **Validación cruzada**: suma de adiciones ≠ `valor_actual - valor_inicial`
  cuando Fase 1 emita eventos.

## 9. Deuda explícita asumida

- 🟧 `proceso_id` NULL en muchos contratos de la muestra (ventanas BDOS
  disjuntas — [0.2 §5.1](0.2-mapeo-campos.md)). Es comportamiento de la
  muestra acotada, no del modelo. En 0.5 con incremental real, el join
  cierra.
- 🟧 Geografía no resuelta en municipios fuera del seed. Cuantificada en
  `validate-reconstruction.sql §4`. Sin perdida de datos: el texto crudo
  vive en `raw_record.payload` y el reproceso de 0.6 solo necesita expandir
  el seed.
- 🟩 `entidad.nivel_gobierno` queda con el dominio crudo de la fuente
  (D24): `territorial/nacional/corporacion_autonoma/otro`. La distinción
  depto vs municipio se infiere del `geografia_id` resuelto, no del texto.
- 🟩 Cuarentena es append-only: re-correr el transform añade duplicados de
  registros que fallan estructuralmente. Esto es intencional (auditoría de
  cada intento). El conteo de la última corrida se obtiene filtrando por
  `batch_id`.

## 10. Orden de ejecución en una base nueva

Contra **Neon** (producción / dev branch) — `DATABASE_URL` en `.env.local`:

```sh
npm run db:migrate            # 1. esquema (0000 + 0001)
npm run db:seed-geografia     # 2. crosswalk DANE (pre-requisito DURO)
npm run db:load-sample        # 3. capa cruda (idempotente por payload_hash)
npm run db:transform          # 4. capa canónica
psql "$DATABASE_URL" -f scripts/sql/validate-reconstruction.sql   # 5. validación
```

Contra **Postgres local** (offline, sin Neon) — añade `DB_DRIVER=node`:

```sh
createdb hydrostack_dev
export DATABASE_URL="postgres://localhost:5432/hydrostack_dev" DB_DRIVER=node
npm run db:migrate            # drizzle-kit conecta por TCP directo
npm run db:seed-geografia
npm run db:load-sample
npm run db:transform
psql "$DATABASE_URL" -f scripts/sql/validate-reconstruction.sql
```

El camino local requiere `pg` (devDependency) y el branch `DB_DRIVER=node` de
`client.ts`. Es la vía que se usó para la corrida del 2026-06-06.

Con esto cerrado, **0.4 entrega**: la canónica reconstruida campo a campo
desde raw, idempotente, reprocesable, sin red. Listo para 0.5 (ingesta
incremental con watermark de doble pasada, D21).
