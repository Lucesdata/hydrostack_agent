## 1. Entorno Postgres — Neon (serverless)

**Decisión:** Neon Postgres, no Postgres en Docker. Razón: el cliente de datos ya está cableado al driver `@neondatabase/serverless` ([client.ts](../../src/lib/db/client.ts)), que habla por **WebSocket** (soporta transacciones, necesarias para la detección atómica de eventos en 0.4). Un Postgres plano en Docker **no** habla ese protocolo sin un proxy WebSocket intermedio; cambiar a Docker implicaría un driver dual y más piezas móviles para cero beneficio en esta fase.

**Reproducibilidad — dev branch de Neon:**
- Neon ofrece *branches* de base como las de git: una rama `dev` aislada, copy-on-write, gratis, que se crea/destruye en segundos. Es el equivalente reproducible a "levantar Postgres local", sin Docker.
- Versión: Postgres 16 (default de Neon).
- Conexión por variable de entorno `DATABASE_URL` (pooled connection string), nunca hardcodeada. Documentada en [.env.example](../../.env.example).

**Arrancar / parar:** no hay proceso local que arrancar — la base vive en Neon. El "arranque" es tener `DATABASE_URL` en `.env.local`; el "parado" es no usarla. Para aislar pruebas destructivas se crea una branch dedicada y se borra al terminar.

> Alternativa Docker descartada para 0.3 (ver §7). Si en el futuro se quiere desarrollo 100% offline, se añade `docker-compose` + driver `node-postgres` condicionado por entorno, sin tocar el esquema.

## 2. Herramienta de migraciones — Drizzle

**Decisión:** Drizzle Kit (ya configurado en [drizzle.config.ts](../../drizzle.config.ts)).

**Por qué Drizzle y no node-pg-migrate ni Prisma**, dado que la capa cruda es JSONB-pesada y habrá transformaciones:
- El esquema se declara en **TypeScript** ([src/lib/db/schema/](../../src/lib/db/schema/)) y es el mismo objeto que consume la capa de transformación (0.4) con tipos inferidos. Una sola fuente de verdad tipada: el `payload jsonb`, los `numeric(20,2)`, las fechas, todo tipado en TS sin un cliente generado aparte (como haría Prisma).
- `drizzle-kit generate` produce **SQL plano versionado** en `drizzle/` — auditable, revisable en PR, aplicable con `drizzle-kit migrate`. No es una caja negra.
- Sin runtime pesado: para queries JSONB crudas se baja a SQL (`sql\`...\``) cuando hace falta, sin pelear con un ORM opinado.

**Versionado, idempotencia, rollback:**
- Migraciones versionadas: `drizzle/0000_clumsy_silver_sable.sql` + journal en `drizzle/meta/`.
- `drizzle-kit migrate` es idempotente: aplica solo las migraciones no registradas en la tabla `__drizzle_migrations`.
- Rollback: Drizzle no genera el `down` automático. Para 0.3 el rollback de la capa cruda es trivial y reversible (`DROP TABLE raw_record CASCADE` documentado); para fases con datos vivos se añadirá un `.down.sql` por migración o se revierte vía branch de Neon (descartar la branch = rollback instantáneo).

**Comandos** (en package.json):
```
npm run db:generate     # regenera SQL desde el schema TS
npm run db:migrate      # aplica migraciones pendientes (requiere DATABASE_URL)
npm run db:studio       # explorador visual
npm run db:load-sample  # carga la muestra de 0.2 a raw_record (0.3)
```

## 3. Tabla cruda `raw_record` — auditoría contra 0.3

La tabla ya existe ([schema/raw.ts](../../src/lib/db/schema/raw.ts)). Auditada contra los requisitos del entregable 3:

| Requisito 0.3 | Columna real | ✓ |
|---|---|---|
| payload completo original en JSONB | `payload jsonb NOT NULL` (sin recortar) | ✅ |
| source / dataset de origen | `source text` (`secop_ii_procesos` \| `secop_ii_contratos`) | ✅ |
| id natural del registro en la fuente | `source_record_id text` (`id_del_proceso` / `id_contrato`) | ✅ |
| hash para detectar cambios | `payload_hash text` (SHA-256 del JSON canónico menos volátiles, D10) | ✅ |
| ingested_at | `ingested_at timestamptz DEFAULT now()` | ✅ |
| (extra) watermark de la fuente | `source_updated_at timestamptz` (nullable) | ✅ |
| (extra) agrupador de carga | `batch_id uuid` | ✅ |

**Índices** (los cuatro pedidos por 0.3, ya presentes):
- `raw_record_payload_gin_idx` — **GIN sobre `payload`** (consultas JSONB exploratorias / containment).
- `raw_record_source_recid_updated_idx` — `(source, source_record_id, source_updated_at)` (histórico de un registro).
- `raw_record_source_hash_idx` — `(source, payload_hash)` (dedupe por hash).
- `raw_record_batch_idx` — `(batch_id)`.

### Estrategia de versionado: **append-only** (decisión clave)

`raw_record` es **append-only e inmutable**: nunca `UPDATE` ni `DELETE`. Cada vez que un registro de la fuente cambia, se inserta una **fila nueva** (un snapshot más), conservando el histórico completo.

**Por qué append-only y no sobrescribir el último estado:** el modelo de eventos de 0.1 (§2) detecta adiciones y prórrogas **comparando el snapshot nuevo contra el anterior**. Si sobrescribiéramos, perderíamos el "antes" y no habría diff posible — el log de `contrato_evento` quedaría ciego. Append-only es el sustrato que hace viable la detección de eventos en 0.4 sin volver a descargar nada.

La idempotencia **no** se logra sobrescribiendo, sino **no insertando** cuando el último snapshot guardado de ese registro ya tiene el mismo `payload_hash` (§4).

## 4. Script de carga — `npm run db:load-sample`

[scripts/load-sample-raw.ts](../../scripts/load-sample-raw.ts) toma la muestra de 0.2 (`samples/procesos.json`, `samples/contratos.json`) y la inserta en `raw_record`. **Reusa el conector de producción** sin reescribirlo:
- `toRawRecord()` ([mapRecord.ts](../../src/lib/ingest/mapRecord.ts)) mapea cada fila SODA → shape de inserción (calcula `payload_hash` con la lógica de `hash.ts`, deriva `source_updated_at`).
- `makeDbSink()` ([dbIngest.ts](../../src/lib/ingest/dbIngest.ts)) hace el insert idempotente.

### Idempotencia y comportamiento ante cambios

El sink consulta el **último** `payload_hash` guardado de cada `source_record_id` y decide:

| Situación | Acción |
|---|---|
| El registro no existe en `raw_record` | inserta (primer snapshot) |
| Existe y el último hash **es igual** | **no inserta** (idempotente: re-correr la carga no duplica) |
| Existe y el último hash **cambió** | inserta **fila nueva** (snapshot histórico; el viejo se conserva) |

Correr `db:load-sample` dos veces seguidas sobre la misma muestra: la segunda corrida inserta **0 filas**. Si la muestra se re-descarga de Socrata y algún registro cambió, esa fila —y solo esa— entra como snapshot nuevo.

**Verificación sin base (`--dry-run`):** mapea las 500+500 filas e imprime un ejemplo (recId + hash + watermark) sin tocar la DB. Ejecutado el 2026-06-05:
```
[dry-run] secop_ii_procesos: 500 filas mapeadas
[dry-run]   ejemplo → recId=CO1.REQ.10002742 hash=6a760ba95512b61f… updatedAt=null
[dry-run] secop_ii_contratos: 500 filas mapeadas
[dry-run]   ejemplo → recId=CO1.PCCNTR.10001 hash=4edc5d436651f435… updatedAt=null
```
(`updatedAt=null` en la primera fila es el artefacto del `$order DESC` documentado en 0.2 §0 — la fila no se descarta, queda sin timestamp.)

> **Nota sobre idempotencia y el caso A→B→A:** el dedupe compara contra el *último* snapshot, no contra todo el histórico. Un registro que cambia y vuelve a un estado anterior (mismo hash que dos snapshots atrás) **sí** generaría un snapshot nuevo — correcto, porque la transición B→A es un evento real. Para la carga única de 0.3 esto no aparece; es comportamiento esperado para el incremental de 0.5.

## 5. Prueba de reproceso

[scripts/sql/reprocess-demo.sql](../../scripts/sql/reprocess-demo.sql) demuestra que desde `raw_record` se reconstruye información **sin volver a Socrata**. Se corre con:
```
psql "$DATABASE_URL" -f scripts/sql/reprocess-demo.sql
```

Incluye 5 consultas:
1. **Inventario** — snapshots y registros distintos por fuente.
2. **Reconstrucción canónica parcial** de contratos (`id`, portafolio, entidad, valor, estado, objeto) extraída del `payload` — el mismo mapeo que hará 0.4, aquí sin red.
3. **Containment JSONB** (`payload @> '{...}'`) — acelerable por el índice GIN.
4. **Filtro sectorial agua** por texto del objeto (scan; el clasificador con score real es 0.2.1 → 0.4).
5. **Idempotencia / append-only** — registros con más de un snapshot (0 tras una sola carga).

Que la reconstrucción (consulta 2) salga **enteramente del `payload` JSONB** es la prueba de que la capa cruda es autónoma: reprocesar = re-leer `raw_record`, nunca re-descargar.

## 7. Decisiones abiertas — con recomendación

### A. Estrategia de versionado de la capa cruda — **append-only (recomendado, adoptado)**
**Opciones:** (a) append-only: cada cambio = fila nueva, histórico de snapshots; (b) sobrescribir el último estado.
**Recomendación: (a).** Es la única coherente con el modelo de eventos de 0.1 §2: el diff de adiciones/prórrogas necesita el snapshot anterior. (b) ahorraría espacio pero cegaría la detección de eventos y violaría D3 (`raw_record` inmutable). Adoptada.

### B. Herramienta de migraciones — **Drizzle (recomendado, ya en uso)**
**Opciones:** node-pg-migrate (SQL plano) · Drizzle · Prisma.
**Recomendación: Drizzle.** Schema TS = fuente de verdad tipada compartida con la transformación 0.4; genera SQL versionado auditable; sin cliente generado aparte (Prisma) ni divorcio schema/tipos (node-pg-migrate). Ya configurado.

### C. Postgres — **Neon dev branch (recomendado, elegido)**
**Opciones:** Docker local · Neon dev branch · Neon directo.
**Recomendación: Neon dev branch.** Reproducible (branch copy-on-write efímera), compatible con el driver serverless actual sin tocar `client.ts`, cero Docker. Docker exigiría driver dual por el protocolo WebSocket de Neon.

- 🟩 **Ingesta incremental / watermark / `sync_log`**: ya existe en `src/lib/ingest/` pero pertenece a **0.5**. En 0.3 la carga es manual y completa sobre la muestra; no se usa el watermark.

## 9. Checkpoint

**¿Puedo reprocesar la información SIN volver a descargar de la fuente?**
Sí. `raw_record.payload` guarda el registro original completo en JSONB. La reconstrucción canónica (consulta 2 de §5) sale enteramente del payload ya aterrizado. Reprocesar = re-leer la capa cruda; nunca se vuelve a llamar a Socrata.

**¿La estrategia de versionado soporta el diff de snapshots del modelo de eventos de 0.1?**
Sí. `raw_record` es append-only: cada cambio de un registro es una fila nueva, conservando el snapshot anterior. La detección de eventos de 0.4 comparará el snapshot nuevo contra el previo del mismo `source_record_id` (índice `source_recid_updated`). Sobrescribir habría hecho esto imposible.

**¿La carga es idempotente?**
Sí. El sink no inserta cuando el último `payload_hash` del registro coincide; re-correr la carga sobre la misma muestra inserta 0 filas. Un registro que cambió respecto a una carga previa entra como snapshot nuevo (no se pierde ni se duplica el no-op).
