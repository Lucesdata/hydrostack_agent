# Fase 0.2 — Spec de ingesta SECOP II

> Documento de diseño. **No es implementación**. No contiene código de ingesta, scripts ni migraciones. Los patrones de query SODA se muestran como contrato de diseño, no como código ejecutable. La implementación corresponde a la sub-fase 0.3.
>
> Depende de [0.1-modelo-datos.md](0.1-modelo-datos.md). Las decisiones D1–D10 se asumen cerradas.

## 0. Contexto y alcance

Esta sub-fase especifica **cómo entran los datos de SECOP II a la capa cruda y cómo se mapean a la capa canónica**. Cubre cuatro cosas:

1. Inventario de los dos datasets SODA (Procesos y Contratos).
2. Hallazgos del esquema real que afectan al modelo 0.1.
3. Mapeo campo-a-campo `payload` → tablas canónicas.
4. Diseño del `payload_hash` (campos volátiles a excluir, D10) y del watermark incremental.

> **Nota de método.** El inventario se levantó contra el endpoint público real (`datos.gov.co`) muestreando filas. Socrata **omite las claves cuyo valor es null** en cada fila, así que una muestra puede no traer todos los campos. Antes de 0.3 hay que cruzar este inventario con el endpoint de metadatos de columnas (`/api/views/{id}/columns.json`) para capturar campos que vinieron nulos en la muestra.

## 2. Hallazgos del esquema real que afectan al modelo 0.1

Tres cosas que el esquema real contradice respecto a lo que asumimos en 0.1. Se documentan aquí porque cambian decisiones, no solo mapeos.

### H1. La llave de join proceso ↔ contrato NO es el ID de proceso

Lo intuitivo sería unir `contrato` con `proceso` por el `id_del_proceso` (`CO1.REQ.xxxx`). **Es incorrecto.** El campo que enlaza es:

```
  contratos.proceso_de_compra   (CO1.BDOS.xxxx)
        ═══ se une contra ═══
  procesos.id_del_portafolio    (CO1.BDOS.xxxx)
```

Es decir, el contrato apunta al **portafolio/BDOS** del proceso, no a su `id_del_proceso` (`CO1.REQ`). Si se hubiera cableado el FK por `id_del_proceso`, el join habría salido vacío de forma silenciosa. `proceso.secop_proceso_id` debe almacenar **ambos** identificadores: el `CO1.REQ` como ID nativo y el `CO1.BDOS` como llave de enlace. → genera **decisión D11**.

### H2. La geografía NO viene como código DANE/DIVIPOLA

El modelo 0.1 (`geografia`, decisión D7) asumió `codigo_divipola` como PK natural. **Ninguno de los dos datasets trae el código DIVIPOLA.** Vienen nombres de texto:

- Procesos: `departamento_entidad` = `"Distrito Capital de Bogotá"`, `ciudad_entidad` = `"Bogotá"`.
- Contratos: `departamento` = `"Distrito Capital de Bogotá"`, `ciudad` = `"Bogotá"`, `localizaci_n` = `"Colombia, Bogotá, Bogotá"`.

Por tanto hace falta una **tabla de mapeo nombre → DIVIPOLA** (un crosswalk DANE) y un paso de resolución durante la transformación. El texto de la fuente es inconsistente ("Bogotá D.C.", "Distrito Capital de Bogotá", tildes, mayúsculas). → genera **decisión D12**. La PK `codigo_divipola` de D7 sigue siendo válida; lo que cambia es que la resolución es un paso explícito con tasa de match medible, no una lectura directa.

### H3. Boolean y nulos vienen como texto centinela

- Booleans llegan como `"No"` / `"Si"` (a veces `"No"`/`"Yes"`), no como `true`/`false`.
- Ausencia de dato llega como centinelas: `"No Definido"`, `"No definido"`, `"No Adjudicado"`, `"Sin Descripcion"`, `"No aplica"`, `"Por definir"`.

Esto confirma la política de basura de D3 (proveedor) y la extiende a **todos** los campos: la transformación necesita un normalizador de centinelas que mapee este conjunto a `NULL` antes de tipar. Un proceso no adjudicado trae `nit_del_proveedor_adjudicado = "No Definido"` → debe resolverse a `proveedor_id = NULL`, no crear un proveedor llamado "No Definido".

## 4. Detección de eventos en ingesta (gancho con 0.1 §2)

Esta sub-fase no especifica el algoritmo de diff completo (eso es 0.2-detección si se separa), pero sí fija de qué campos del contrato salen los eventos:

| Cambio entre snapshots | Campo SODA observado | `tipo_evento` |
|---|---|---|
| sube `valor_del_contrato` | `valor_del_contrato` | `adicion` |
| se mueve `fecha_de_fin_del_contrato` | `fecha_de_fin_del_contrato` (+ `dias_adicionados`) | `prorroga` |
| `estado_contrato` → suspendido | `estado_contrato` | `suspension` |
| `estado_contrato` → terminado/cerrado | `estado_contrato` | `terminacion` |
| cambia `documento_proveedor` | `documento_proveedor` | `cesion` |

El watermark (§6) garantiza que veamos los snapshots nuevos; el `payload_hash` (§5) garantiza que solo procesemos los que cambiaron de verdad.

## 5. `payload_hash` — campos volátiles a excluir (concreta D10)

D10 decidió excluir del hash los campos volátiles para no generar eventos espurios. Con el esquema real, la lista de exclusión por dataset es:

**Contratos (`jbjy-vk9h`):**
- `ultima_actualizacion` — cambia en cada republicación aunque nada sustantivo cambie. **Excluir siempre.**
- Campos de ejecución financiera que la entidad actualiza sin que sea modificación contractual: `valor_facturado`, `valor_pagado`, `valor_pendiente_de_pago`, `valor_amortizado`, `valor_pendiente_de_ejecucion`, `saldo_cdp`, `saldo_vigencia`. → **Decisión D13**: ¿estos cambios merecen evento propio (`ejecucion`) o solo actualizan la foto sin generar evento? (recomendación abajo).

**Procesos (`p6dx-8zbt`):**
- `fecha_de_ultima_publicaci` — watermark, excluir.
- Contadores de actividad que cambian sin ser cambio sustantivo del proceso: `visualizaciones_del`, `proveedores_invitados`, `proveedores_con_invitacion`, `respuestas_al_procedimiento`, `respuestas_externas`, etc. **Excluir del hash.**

El hash se calcula sobre el JSON canónico (claves ordenadas) **menos** estos campos. La lista exacta y completa se congela al cruzar con el endpoint de columnas antes de 0.3.

## 6. Diseño del watermark incremental

### 6.1 Por fuente, no global

Cada dataset tiene su propio campo de watermark (§1) y su propia fila de control en `sync_log` con `source` distinto. El incremental de Procesos y el de Contratos avanzan independientes.

### 6.2 Query de extracción (contrato de diseño, no código)

Paginación **keyset** sobre el campo watermark, no `$offset` (que en Socrata se degrada y puede saltar filas en datasets grandes):

```
GET /resource/jbjy-vk9h.json
  ?$where = ultima_actualizacion > '{watermark_anterior}'
  &$order = ultima_actualizacion ASC, id_contrato ASC
  &$limit = {page_size}
```

- `watermark_anterior` = `MAX(watermark_to)` de `sync_log` para esa fuente con `status IN ('ok','partial')` (0.1 §4.5).
- Tiebreaker `id_contrato ASC` para estabilidad cuando varios registros comparten timestamp.
- Cada página se vuelca a `raw_record` (append-only) antes de transformar.
- Al cerrar la corrida, `sync_log.watermark_to` = el mayor `ultima_actualizacion` visto.

### 6.3 Backfill histórico (concreta D9)

Backfill en ventanas trimestrales por el campo watermark, cada ventana con su `batch_id`. Permite reintentar un trimestre sin reprocesar el resto. El incremental arranca donde termina el backfill.

### 6.4 Frontera de relojes

El `>` estricto puede perder registros que compartan exactamente el timestamp del watermark con otros ya procesados. Mitigación de diseño: solapar la ventana restando un margen (p. ej. `watermark_anterior - 1 día`) y dejar que el `payload_hash` descarte los duplicados sin cambios. El costo es reprocesar un margen pequeño; el beneficio es no perder filas. → **Decisión D14**.

## 7. Decisiones nuevas (confirmadas 2026-06-04)

Surgieron del esquema real. Las cinco se cierran con la recomendación marcada.

### D11. Identificador de enlace proceso↔contrato — **confirmada**
`proceso` guarda el `CO1.REQ` (nativo) y el `CO1.BDOS` (`portafolio_id`, llave de enlace con `contrato.proceso_de_compra`).
**Decisión:** persistir ambos. `secop_proceso_id` = `CO1.REQ` como ID nativo; `portafolio_id` = `CO1.BDOS` con índice, usado para resolver el FK desde contrato. La resolución puede quedar pendiente (`contrato.proceso_id = NULL`) si el proceso aún no se ha ingerido — se resuelve en una pasada posterior, no se descarta el contrato.

### D12. Resolución de geografía texto → DIVIPOLA — **confirmada**
La fuente da nombres, no códigos.
**Decisión:** tabla `geografia` con `codigo_divipola` PK (D7 intacto) + tabla puente `geografia_alias` (texto normalizado → `codigo_divipola`) sembrada con el crosswalk oficial DANE. La transformación normaliza el texto (minúsculas, sin tildes) y busca en alias. Match fallido → `geografia_id = NULL` + guardar el texto crudo en `contrato`/`proceso` para medir tasa de match y ampliar el alias. No bloquear la ingesta por geografía no resuelta.

### D13. Cambios de ejecución financiera: ¿evento o solo foto? — **confirmada (solo foto)**
`valor_facturado`, `valor_pagado`, etc. cambian seguido sin ser modificación contractual.
**Decisión:** **no** generar `contrato_evento` por ellos en fase 0 (excluidos del hash, §5). Actualizar solo la foto. Si más adelante se quiere serie de ejecución, va a una tabla derivada propia (`contrato_ejecucion_snapshot`), no al log de eventos contractuales — son cosas distintas y mezclarlas ensucia el log.

### D14. Margen de solape del watermark — **confirmada**
**Decisión:** restar 1 día al watermark anterior en cada corrida y confiar en `payload_hash` para deduplicar. Simple, robusto frente a registros con el mismo timestamp. Costo de reproceso despreciable.

### D15. Filtro sectorial agua/saneamiento — ¿en ingesta o post-ingesta? — **confirmada (post-ingesta)**
HydroStack es vertical (agua/saneamiento), pero los datasets son de todo el Estado.
**Decisión:** ingerir **todo** a `raw_record` sin filtrar (ELT puro: filtrar en extracción rompe la posibilidad de reclasificar después), y aplicar el filtro sectorial en la capa canónica o derivada usando `codigo_principal_de_categoria` (UNSPSC), `sector`, `tipo_de_contrato` y matching de `objeto`. El criterio exacto de "esto es agua/saneamiento" es una decisión de producto que merece su propio mini-spec (pendiente, fuera de 0.2).

## 8. Checkpoint

**Pregunta:** ¿el spec de ingesta permite poblar la capa cruda y derivar la canónica de 0.1 sin contradecir el modelo, y deja el incremental listo para diseñarse en detalle?

**Respuesta:** sí, con tres ajustes al modelo 0.1 que NO son rediseño estructural:

1. **`proceso` gana `portafolio_id`** (CO1.BDOS) como llave de enlace (D11). Es una columna nueva, no un rediseño.
2. **`geografia` gana tabla puente `geografia_alias`** para resolver texto → DIVIPOLA (D12). La PK `codigo_divipola` de D7 no cambia.
3. **Normalizador de centinelas** (`"No Definido"` → NULL) como paso previo a tipar, válido para todos los campos (H3). Es lógica de transformación, no esquema.

Lo que el modelo 0.1 ya soportaba y se confirma con datos reales:
- El watermark por fuente (`sync_log`) calza con que cada dataset tiene su propio campo de actualización.
- La separación estado/eventos calza con que `valor_del_contrato`, `fecha_de_fin_del_contrato` y `estado_contrato` son justo los campos que mutan y disparan eventos.
- La política de basura de NIT (D3) calza con los centinelas reales observados.

**Decisiones D11–D15 cerradas el 2026-06-04.** Cambios al modelo 0.1 que se arrastran a 0.3:
- `proceso.portafolio_id` (CO1.BDOS), indexado (D11).
- Tabla puente `geografia_alias` (texto normalizado → `codigo_divipola`) (D12).
- Campos de ejecución (`valor_facturado`, `valor_pagado`, ...) solo actualizan la foto, no generan evento (D13).
- Margen de solape de 1 día en el watermark (D14).
- Sin filtro sectorial en ingesta; clasificación agua/saneamiento en capa derivada (D15).

**Pendiente fuera de 0.2:** mini-spec del criterio de clasificación sectorial agua/saneamiento (D15). Spec de ingesta listo para entrar a 0.3 (implementación / DDL).
